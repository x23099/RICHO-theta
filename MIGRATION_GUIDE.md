# 別PCでの実験に向けた移行・環境構築ガイド

このドキュメントは、RICOH THETAカメラの送受信プログラム（映像配信およびUI表示）を、開発PCから別の実験用PCへ移行する際に発生しうるエラーと、その対処手順をまとめたものです。

---

## 1. 依存ライブラリの不足（Pythonパッケージ）

別PCの仮想環境（`theta-env`）が初期状態の場合、必要なPythonパッケージが不足しており `ModuleNotFoundError` が発生する可能性が非常に高いです。

### 必要な追加パッケージ一覧
* **GUI / 画像処理**: `PySide6`, `opencv-python`, `numpy`
* **WebRTCストリーミング**: `aiortc`, `aiohttp`, `av` (PyAV)
* **コントローラー入力（オプション）**: `evdev`

### 対処手順
移行先PCの仮想環境をアクティベートするか、仮想環境の `pip` を直接指定して以下のコマンドを実行してください。

```bash
# 仮想環境のパスが /home/robo25/theta-env の場合
/home/robo25/theta-env/bin/pip install PySide6 opencv-python numpy aiortc aiohttp av evdev
```

> [!NOTE]
> `av` (PyAV) のインストールには、システム側の FFmpeg 開発ライブラリが必要になる場合があります。インストール時にエラーが出る場合は、先にシステムパッケージ（後述）をインストールしてください。

---

## 2. システムパッケージの不足（APT）

ストリーミングの起動スクリプト（`start_theta_stream_rtc.sh`）は、いくつかのLinuxシステムコマンドに依存しています。

### 依存するシステムツール
* `xdotool`: UIウィンドウの自動検索およびウィンドウ解像度の取得に使用します。
* `ffmpeg`: WebRTCのキャプチャ（`x11grab`）のバックエンドとして動作します。

### 対処手順
以下のコマンドを実行して、必要なシステムパッケージをインストールしてください。

```bash
sudo apt update
sudo apt install -y xdotool ffmpeg
```

---

## 3. ROS 2 環境へのアクセスエラー

仮想環境 (`theta-env`) を使用して python スクリプトを実行すると、ROS 2 のシステムパッケージやカスタムメッセージのパスが引き継がれず、`rclpy` や `kobuki_ros_interfaces` などのインポートエラーが発生することがあります。

### 対処手順
起動スクリプトを実行する前に、ホストOS側で ROS 2 のセットアップと、カスタムワークスペースのセットアップを必ず行ってください。

```bash
# ROS 2のセットアップ（インストールされているディストリビューション名に合わせてください）
source /opt/ros/<ros_distro>/setup.bash

# コブキなどのカスタムメッセージ用ワークスペースのセットアップ
# 例: kobuki_ws がある場合
source /home/robo25/kobuki_ws/install/setup.bash
```

## 4. カメラデバイスファイル（`/dev/video*`）の権限・パス違い

起動スクリプトは、デフォルトでカメラデバイスとして `/dev/video0` を参照しますが、第二引数からデバイスファイルを指定できます。

### 対処手順
1. **デバイスの確認**
   THETAカメラを接続した状態で以下のコマンドを実行し、認識されているビデオデバイスを確認します（例: `/dev/video2` など）。
   ```bash
   ls -la /dev/video*
   ```
2. **権限の付与**
   デバイスファイルに書き込み/読み込み権限がない場合は、パーミッションを変更するか、実行ユーザーを `video` グループに追加します。
   ```bash
   sudo usermod -aG video $USER
   # 反映のために一度ログアウトして再ログインするか、一時的に以下を実行
   sudo chmod 666 /dev/video2
   ```
3. **起動時のデバイス指定**
   スクリプトの第二引数に対象デバイスを指定して起動します。
   ```bash
   # 受信側IPが150.89.169.70、デバイスが/dev/video2の場合
   ./start_theta_stream_rtc.sh 150.89.169.70 /dev/video2
   ```

## 5. DISPLAY環境変数のエラー

別PCにSSH等でリモート接続してスクリプトを実行する場合、`QXcbConnection: Could not connect to display` や `x11grab` のオープンエラーが発生します。

### 対処手順
スクリプトを実行する前に、接続先PCのディスプレイ設定に合わせた `DISPLAY` 環境変数を設定してください。
```bash
# 通常は :0 または :1.0 です
export DISPLAY=:0
```
また、[start_theta_stream_rtc.sh](file:///home/robo25/theta_ws/RICHO-theta/src/start_theta_stream_rtc.sh) 内の `DISPLAY_ID` も環境に合わせて必要に応じて書き換えてください。

## 6. 小さな画面解像度（1024x768等）への対応

X11の仕様上、`x11grab`（WebRTCでウィンドウをキャプチャして送信する機能）は**画面からはみ出している部分や他のウィンドウの裏に隠れている部分を正常にキャプチャできません**（黒塗りになったり表示が崩れたりします）。

移行先PCの画面解像度が `1024x768` と小さいため、従来のUI（最小サイズ `1450x1080`）のままだと画面外にはみ出してしまい、キャプチャされたストリーム映像が見切れてしまいます。

### 対処手順
従来の大きな表示サイズを維持しつつ、小さいモニターでも見切れずに起動できるよう、切替オプションを導入しました。

1. **最小ウィンドウ制限の縮小（共通）**: `zc33s_ui.py` 内の最小サイズ制限を `1450x1080` から `900x700` に引き下げました。これは**最小サイズ制限**であるため、大きなモニターでは従来通り手動で大きくウィンドウを広げることができ、元の表示サイズ（最大化や拡大）には悪影響を及ぼしません。
2. **レイアウトサイズ切替オプションの追加**: [start_theta_stream_rtc.sh](file:///home/robo25/theta_ws/RICHO-theta/src/start_theta_stream_rtc.sh) の**第三引数**に `small`（または `1024`）を指定することで、各ビューの大きさを動的に小さく起動できるようになりました。
   * **これまでの実験PC（大画面）で起動する場合（デフォルト）**:
     ```bash
     # 第3引数を省略するか、largeを指定すると、従来の大画面レイアウトで起動します
     ./start_theta_stream_rtc.sh 150.89.169.70 /dev/video0
     ```
   * **移行先の1024x768モニターで起動する場合**:
     ```bash
     # 第3引数に "small" を指定すると、UIが800x450フロント等の小さなレイアウトで収まります
     ./start_theta_stream_rtc.sh 150.89.169.70 /dev/video2 small
     ```
     ※UI全体が1024x768の画面内にスッポリ収まるため、`x11grab` でも全体を完璧にキャプチャして受信側に届けることができます。
