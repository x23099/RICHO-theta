# WebRTCカメラ映像・制御データの低遅延化プロセス報告書（引き継ぎ用）

本ドキュメントは、RICOH THETAを用いたWebRTC配信システムにおいて、映像および車両テレメトリデータの遅延（ラグ）を極小化するために行ったボトルネック分析、設計変更、および実装プロセスの記録です。別PCへの展開や今後の開発の引き継ぎにご活用ください。

---

## 1. 背景と課題

* **初期状態**: 映像の遅延（ラグ）が非常に大きく、車両の遠隔操作に支障をきたすレベルであった。
* **中間状態**: 各種パラメータ調整とソケット最適化により、遅延は一時 **328 ms** に。
* **最終状態**: 送受信のアーキテクチャ刷新（インプロセス化・スレッド分離・DataChannel化）により、制御テレメトリ遅延は **約2.5 ms**、映像を含めた総合ラグは **283 ms** まで削減を達成。

---

## 2. 検出されたボトルネックと分析

システムを詳細に分析した結果、以下の4つの要因が遅延を累積させていることが判明しました。

1. **GUIスレッドの同期ブロッキング (`self.grab()`)**
   * 送信側で画面全体を `self.grab()` してラスタライズし、JPEG圧縮する処理がメインGUIスレッド上で同期的に実行されていたため、フレーム処理ごとに数十ミリ秒のブロッキングが発生していました。
2. **プロセス間通信のオーバーヘッド**
   * UI画面（`zc33s_ui.py`）と配信プロセス（`webrtc_stream.py`）が別プロセスで動き、ローカルソケット（TCP:9999）を介してJPEGデータをシリアライズ・転送・デシリアライズしていたため、メモリコピーと通信遅延がチリツモで増加していました。
3. **`cap.read()` による asyncio イベントループのフリーズ**
   * aiortcの非同期送信メソッド `recv()` 内で、カメラ画像をキャプチャする `cap.read()` を直接（同期的に）呼んでいたため、画像の取り込みが完了するまでWebRTCのイベントループ全体が毎フレームフリーズしていました。
4. **受信側イベントキューの詰まり（遅延の自己増殖）**
   * 受信側で映像パケットが届くたびに直接描画（`remap`とGUI反映）を呼び出していたため、PC of 操縦席の描画負荷によって一時的にGUIが遅れると、古い描画イベントがキューに溜まり続け、時間経過とともに遅延が無限に蓄積していました。

## 3. 実施した具体的な解決策

### 💡 ゼロコピー ＆ DataChannel 統合アーキテクチャへの刷新

```mermaid
graph TD
    subgraph 改善前 (ソケット通信 & 同期キャプチャ)
        A1[THETAカメラ] -->|同期 read| B1[birds_eye_view_ui.py UI]
        B1 -->|grab & JPEG圧縮| C1(ImageSocketServer:9999)
        C1 -->|TCPソケット通信| D1[webrtc_stream.py 配信]
        D1 -->|VP8エンコード| E1[WebRTC Stream]
    end

    subgraph 改善後 (極限ゼロコピー & 非同期スレッド & DataChannel)
        A2[THETAカメラ] -->|バックグラウンドスレッドで常時読込| B2[webrtc_stream.py]
        B2 -->|メモリポインタ直接渡し| C2[av.VideoFrame.from_ndarray]
        B2 -->|VP8エンコード (realtime)| D2[WebRTC Video Stream]
        
        ROS[ROS 2 /odom等] -->|インプロセス購読| B2
        B2 -->|JSONテキスト| E2[WebRTC DataChannel]
        
        D2 -->|生の魚眼映像| F2[webrtc_receive_ui.py]
        E2 -->|車両データ| F2
        
        F2 -->|タイマー駆動 41ms で最新フレームをプルして描画| G2[低遅延表示画面]
    end
```

### 1. 送信側のインプロセス化と極限ゼロコピー化
* **対策**: 送信側のGUI表示を完全に廃止し、カメラから取得した画像をメモリ上で直接 `av.VideoFrame.from_ndarray` に変換して WebRTC の VideoTrack に手渡すように `webrtc_stream.py` に一本化しました。ソケット通信およびJPEGの圧縮・解凍処理が消滅し、メモリコピーが最小限になりました。
* **エンコーダ最適化**: VP8エンコーダに対して `deadline="realtime"`（リアルタイムモード）、`lag-in-frames="0"`（フレーム先読みゼロ）を適用しました。

### 2. WebRTC DataChannel による車両データ伝送 (ステップ①)
* **対策**: メーター描画（`cv.remap`や予測線計算）を送信側から受信側へと完全移植しました。速度、ギア、オドメトリ、バッテリーなどの車両データは、WebRTC の **DataChannel (名称: telemetry)** を介してJSONテキストで伝送し、受信側でメータを自機描画するようにしました（制御遅延が **約2.5ms** に）。

### 3. 送信側キャプチャの別スレッド化（非ブロッキング化）
* **対策**: `cv2.VideoCapture.read()` を別スレッド（`_capture_loop`）で常時ループさせ、最新フレームを変数に格納し続けるようにしました。WebRTCの `recv()` はこのメモリ変数を即座に読み出すだけにしたため、asyncio のイベントループのフリーズが完全に解消されました。

### 4. 受信側描画のタイマー駆動化（フレームドロップ）
* **対策**: 映像受信スレッドと描画処理を分離しました。受信スレッドは届いた画像を最新の1枚に上書き保存するだけに留め、GUIスレッド側は `QTimer`（24FPS = 41ms周期）で「その瞬間の最新フレーム」を引っ張ってきて描画する仕組み（Pull方式）にしました。これにより、負荷の変動があってもキューが一切詰まらず、遅延が蓄積しないシステムになりました。

### 5. 垂直同期（V-Sync）の無効化
* **対策**: 受信側UI起動時に `swapInterval = 0` をコンテキストに設定し、GPUでのレンダリングが完了した瞬間にウェイトなしで画面を書き換えるようにしました。

---

## 4. 変更ファイルとリポジトリ構成 (`RICHO-theta/src/`)

* **`webrtc_stream.py`** (上書き): 送信側。非ブロッキングキャプチャスレッド、ROS 2テレメトリ購読、およびWebRTC DataChannel送信を行います。
* **`webrtc_receive_ui.py`** (新規): 受信側。WebRTC映像およびDataChannelを受信し、GPU V-Sync無効化の状態で魚眼展開（フロント/BEV）、走行予測線、ダッシュボードメーターを自機描画します。
* **`start_theta_stream_rtc.sh`** (上書き): 送信側の起動用。従来通りのコマンド引数でインプロセス配信スクリプトを直接実行します。
* **`start_theta_receive_rtc.sh`** (新規): 受信側の起動用。適切な解像度・視野角設定で受信用UIを起動します。

---

## 5. 別PCでの実行手順

### ① 最新コードの取得（双方のPC）
```bash
git pull
```

### ② 送信用PC（ロボット側）での起動
今までのコマンドでそのまま実行可能です（UIやx11grabは立ち上がらず、超軽量なCLI配信ノードとして機能します）。
```bash
./src/start_theta_stream_rtc.sh [受信側IP] /dev/video0
```

### ③ 受信用PC（操縦側）での起動
G923ハンコンを接続した状態で、受信用スクリプトを起動します。
```bash
./src/start_theta_receive_rtc.sh
```
これにより、操縦PC側で自動的に接続が確立され、極小レイテンシで走行映像とメーターが表示されます。
